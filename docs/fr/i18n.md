# Stratégie d’internationalisation

🌍 Langue :
- 🇫🇷 Français
- 🇬🇧 [English](../en/i18n.md)

---

## Objectif

Le projet doit supporter les libellés anglais et français pour toutes les valeurs enum exposées au frontend.

C’est particulièrement important pour une future exploitation via Lovable.

---

## Règle principale

Les enums restent techniques et stables.

Ils sont toujours définis en anglais.

Les traductions sont gérées en dehors du modèle de domaine via des clés i18n stables.

---

## Format des clés

```text
{NomEnum}.{Valeur}

Exemples :

GuardrailSeverity.Warning
DocumentStatus.Indexed
RagQueryStatus.BlockedByGuardrail
Pourquoi ne pas traduire directement les enums ?

Les enums font partie du contrat technique.

Ils ne doivent pas contenir des libellés UI ou des valeurs traduites.

Mauvais exemple :

public enum GuardrailSeverity
{
    Info,
    Avertissement,
    Bloque
}

Bon exemple :

public enum GuardrailSeverity
{
    Info = 0,
    Warning = 1,
    Blocked = 2
}
Contrat API

Quand des valeurs enum sont exposées au frontend, l’API doit retourner :

{
  "value": "Warning",
  "code": 1,
  "i18nKey": "GuardrailSeverity.Warning"
}

Le frontend peut utiliser i18nKey pour afficher le libellé traduit.

Fichiers de traduction MVP

Les fichiers de traduction sont stockés ici :

src/SecureRagAgent.Contracts/Localization/en.json
src/SecureRagAgent.Contracts/Localization/fr.json
Décision de conception

La couche Domain ne doit pas dépendre de la localisation.

La localisation appartient à la couche contrat API/UI
